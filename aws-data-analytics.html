Kinesis

Managed alternative to Kafka – Great for any kind of real time data (like logs, metrics, IoT, clickstream data, etc) – Integrated well with stream processing frameworks like Spark - Data is automatically replicated synchronously to 3AZs.

There are just 2 factors for pricing – and hourly charge per shard and a charge for each 1 million PUT transactions

Do not use it for long term storage – Do not use for really small messages (<200KB)

  • Consists of
  • Kinesis Streams or Kinesis Data Streams
  • Kinesis Firehose
  • Kinesis Analytics

Kinesis Data Streams: utilized for low latency stream ingestion at scale - realtime

  • Streams are divided into Shards (or partitions) – Billed based on number of shards – All messages within a shard are ordered but not globally (i.e across all the shards)

Producers --> 1…n Shards --> Consumers

  • Data Retention is 24 hours by default and can go to a max of 7 days.
  • Once the data has been read it is not deleted from the stream. The data stays for the duration of the Data Retention period. Multiple consumers can consume the data at the same time.
  • Also note that the streams are ‘append only’. i.e. once data has been inserted then it cannot be altered or deleted
  • Kinesis Data Stream Records / messages are made up of
  • Data Blob (bytes – can be anything) – max size 1MB
  • Record Key – which shard to send data to
  • Sequence Number – Unique sequence number generated automatically when every message is added to a shard
  • Kinesis Data Stream Producers:
  • Kinesis Producer SDK
  • Kinesis Producer Library (KPL)
  • Kinesis Agent
  • 3rd party libraries (Spark, Kafka, NiFi, etc)
  • Managed AWS Sources (CloudWatch Logs, AWS IoT, Kinesis Data Analytics)
  • Kinesis Producer SDK: to be utilized for simple – low latency scenarios - If we go above the size / throughput limits (1MB/second or 1000 messages/s at write PER SHARD) then a ProviderThroughputExceed exception is thrown.
  • PutRecord – one record (1 user record / 1 stream record / 1 http call)
  • PutRecords – many records (multiple user records / 1 stream record / I http call) – utilizes batching

Options to try when ProvisionedThroughputExceeded Exception is received 1) ensure you don’t have a hot shard i.e. example if you key is device type and 90% of your devices happens to be iPhones then all messages will end up going to the single shard or 2) Retries with back off or 3) increase number of shards

  • Kinesis Producer Library (KPL): Highly configurable C++ / Java library – Built in retry mechanism – supports both Synchronous and Asynchronous APIs (for better performance) – Built in Batching (Aggregate and Collect) and on by default – Compression not supported and need to be done programmatically

Batching Types: Aggregate and Collect - 1) Aggregate: waits till more user records and aggregates all the info into one stream record (assuming the 1MB limit is not reached) and 2) Collect: when previous aggregation is completed and there are 2 user records instead of calling PutRecord twice PutRecords is invoked (assuming 1MB limit is not reached)

Note: When using PutRecords a single failure does not stop the processing of subsequent records – a HTTP 200 response is still returned – In this case to detect partial failures the ‘PutRecordsResult’ method in the KPL can be utilized to detect individual user record failures and then retry the PUT

Batching interval governed by setting RecordMaxBufferedTime (default: 100ms). Don’t use KPL if you want to send the data immediately to the stream since KPL is asynchronous plus also coz RecordMaxBufferedTime

  • Kinesis Agent: Java based agent built on top of KPL – Can be utilized currently in only Linux environments and primary used to upload log files – basic pre-processing and transformations (csv to json, logs to json, etc) can be done
  • Kinesis Consumers:
  • Kinesis Consumer SDK
  • Kinesis Client Library (KCL)
  • Kinesis Connector Library
  • Kinesis Enhanced Fan Out
  • 3rd party libraries (Spark, Kafka Connect, Log4J, Flume, NiFi, etc)
  • Managed AWS Sources (Kinesis Data Firehose, Kinesis Analytics, Lambda)
  • Kinesis Consumer SDK: Polling mechanism with GetRecords API call - Max throughput is 2MB per second / shard – GetRecords returns upto 10MB of data or upto 10K records – Given the 10MB goes over the 2MB limit 5 seconds will need to be waited before the next call – plus max only 5 GetRecords calls can be made every second…so 200ms latency…so if we have more consumers then they will all share the 2MB size limit and 5 GetRecords calls per second limit (Example -> we have 10 consumers then the latency will be 2 seconds per consumer) . For better results see -> Kinesis Consumer Enhanced Fan Out option
  • Kinesis Client Library (KCL): Java First but also supported for Golang, Python, Ruby, Node, .NET, etc. – In built de-aggregation (for KPL aggregation) turned on by default

Supports checkpoint feature i.e. case of 2 applications points to the same shard and if one of the applications goes down and comes back up then it is able to pick up where it left off – the checkpointing is done by utilizing Dynamo DB in the background (given DynamoDB is utilized ensure enough WCU / RCU is utilized or utilize On-Demand provisioning for Dynamo DB . (Important – if this is not provisioned correctly then this has the potential to slow down consumption)

KCL uses the name of the Streams application to create the DynamoDB table name, so use unique names in KCL. Each row in the DynamoDB represents a shard and the partition/hash key is the shard id.

Default provision through for DynamoDB for checkpointing is 10RCU and 10 WCU

  • Kinesis Connector Library: Older legacy library – Java Based – Used along with Kinesis client libraries – connectors for DynamoDB, RedShift, S3 and Elastic Search– running on EC2 – really not used – same function can be done by Lambda or Firehose. The Connector library uses KCL behind the covers plus add additional functionality for talking to the specific AWS service + transforms / filters / batching
  • Kinesis Enhanced Fan Out: Need to have KCL 2.0 or Lambda Nov 2018 or later – No polling but Push Model – consumer initially SubscribeToShard() and then data is pushed to consumer at the rate of 2MB per second per shard per consumer – More consumers can be added to increase the throughput – max consumers is 5 per shard but this can be increased by contacting AWS support – Latency reduced to ~70ms – catch....this is more expensive!
  • Kinesis Scaling: Adding of shards is called ‘Shard Splitting’ – essentially 2 new shards are created instead of an existing one – the existing shard is available till the time all the messages in the shard are not expired. This can be utilized to rectify a ‘hot’ shard. Opposite of this is ‘Merging Shards’ done in order to decrease capacity and save costs. Merging creates a new shard and the 2 existing shards are around till the data is emptied.

Limitations: Re-sharding cannot be done in parallel – and it takes a few seconds per shard…example doubling shards from 1000 to 2000 takes about 8 hours.

Auto Scaling option exists but not a native feature and needs to be done manually.

  • Kinesis Security: Control access / authorization using IAM policies - Encryption in flight using HTTPS endpoints - Encryption at REST using KMS - Client side encryption can be done (but is harder)

SQS

  • Guarantees at least once delivery – pull based system. Less than 10ms latency
  • scales automatically from 1msg/second to 10kmsgs/second. No limit on number of consumers attached. Max 120,000 inflight messages that can be processed
  • Messages can be kept for a min of 1 minute or a max of 14 days. Default is 4 days.
  • Two Types – Standard Qs (best effort but order is not guaranteed) and FIFO Qs limited to 3000msgs/second (using batching) – De-duplication if Duplication ID is passed within a 5 minute window.
  • There is a risk of duplicate processing here and should be taken care by the app. The ordering is best effort ordering and so there can be out of order messages
  • SQS messages are made up of
  • Message Body (String) – 256KB max message size
  • Message Attributes (key-value pairs) [optional]
  • Delay Delivery [optional]
  • Response – MessageId and MD5 hash of the message
  • Consumption: Poll SQS for messages – max 10 at a time/consumer. Processing of the message needs to occur within the visibility timeout. Long polling is supported i.e. poll less frequently when there are no messages on the queue
  • Visibility Timeout: The amount of time the message is sitting invisibly on the queue while it waits for the message to be fully processed. If the job is not fully processed the message becomes visible again after the visibility timeout period. Max = 12 hours
  • Is message of size more than 256KB has to be sent then SQL Extended Client (Java Library) can be utilized. Beneath the covers the message is stored in S3.
  • Pay per API Request and pay by network usage
  • In flight encryption using HTTPS – Can enable SSE using KMS for encryption at rest (only the message body is encrypted and not the message attributes) – IAM policies can be applied – access policy per SQS queue can be applied
  • can be used with SNS to achieve fan out / async parallel processing

S3

  • Object based storage – stored in buckets a.k.a folders / directories.
  • Buckets must have globally unique name and are defined at a region level – naming convention { no uppercase | no underscore | 3-63 characters long}
  • File sizes can be from 0 to 5TB. Storage is unlimited. If files are above 5GB then utilize multipart upload. The multipart upload does a parallel operation and this also ensures faster performance.
  • ‘Read after Write’ consistency for PUTs (creates) and ‘Eventual’ consistency for overwrites (updates) and deletes. Note: one caveat for PUTs is it is immediately preceded by a GET call -> in this case eventual consistency is seen (think transactions)
  • What is stored: {key, value, version id, metadata, security info}. The key is essentially the full path to the object starting at the bucket level
  • Pattern: https://s3-us-west-2.amazonaws.com/myBucket/myFile.data

Provisioned Throughput

  • When a table is created then read and write capacity units must be specified. Read Capacity Units (RCU) and Write Capacity Units (WCU). The allocated RCUs/WCUs are distributed equally across the available partitions
  • It is an option to not select the units by selecting
  • On Demand option (this is expensive).
  • Auto Scaling option: is similar to On Demand mode but we specify a min and max RCU/WCU values within which scaling occurs
  • Manual: manual
  • When RCUs and WCUs have been specified then throughput can be exceeded by using “burst credit”. If the burst credits are exhausted then “ProvisionedThroughputException” will be received. It is then recommended to do exponential back-off retry
  • 1 WCU represents 1 write per 1 second for an item up to 1KB in size.

Examples:

[1 write every 1s of size 1KB] [111]

10 objects / second of 2KB each => 10 * 2 = 20 WCUs

6 objects / second of 4.5KB each => 6 * 5 = 30 WCUs (KB gets rounded up)

120 objects / minute of 2KB each => 120 / 60 * 2 => 4WCUs

  • 1 RCU represents 1 strongly consistent read up to 4KB per second or 2 eventually consistent reads per second for an item up to 4KB in size.

Examples:

[1 strongly consistent read every 1s of size 4 KB] [114]

[2 eventually consistent read every 1s of size 4 KB] [214]

10 strongly consistent reads / s with 4KB each => 10 * (4 / 4) = 10RCUs

16 eventual consistent reads / s with 12KB each => (16 / 2) * (12 / 4) = 24RCUs

10 strongly consistent reads / s with 6KB each => 10 * (6 / 4) = 20RCUs (rounded)

  • Note- by default DynamoDB follows Eventual Consistency for reads. GetItem, Query and Scan provide a “ConsistentRead” parameter that can be set to true to ensure Strongly Consistent reads.
  • If RCUs and WCU are exceeded then a ProvisionedThroughputException is received. Some of the reasons for this is 1) Hot Keys / Partitions – case where one partition key/keys is read too many times – example a very popular item 2) very large items being read/written. Some solutions are 1) Exponential back-off when exception is encountered (baked into SDK) 2) Distribute partition keys as much as possible 3) Utilize DynamoDB Accelerator (DAX)
  • Utilize DynamoDB Accelerator (DAX): Intermediate cache layer in front of DynamoDB. Writes go right through – Reads go to DAX instead. Default time-to-live 5 mins. Can be up to 10 nodes in the DAX cluster. Min recommended 3 nodes

Partitions

  • Few scenarios for partition key distribution
  • You start with one partition. Each partition can have a max of 3000RCU / 1000WCU and Max of 10GB. After this a new partition will be created (IMPORTANT: Automagically)

To compute number of partitions required –

Find Max of(Ceiling (Total rcu / 3000 + Total wcu / 1000) OR (Total Size / 10GB))

  • Note: that the partitions do not scale down after peak demand times and this can lead to perf issues – i.e. during peak times say you have increased the partitions from 5 to 10. Now during off peak times the WCU/RCUs allocated are across the 10 partitions and not 5 so essentially each partition gets reduced WCU/RCUs

Index

  • Local Secondary Indexes (LSI) and Global Secondary Indexes (GSI)
  • Local Secondary Indexes: Partition key is the same but there is an alternate sort/range key. You can define up to 5 LSIs. Each range/sort key must be one scalar attribute and either String, Number or Binary – LSI can be defined only at table creation time – The LSI sort key may or may not be present…so the actual number of records in the index can be smaller than the number of records in the table...hence called sparse index (and this therby speeds things up as it is a smaller subset of data to look at) – In addition to the partition key + sort key + LSI value additional projection attributes can be added to the index…example result in example below

RCU/WCU is shared with the main table

  • Global Secondary Indexes: In this case the partition key is also new and optionally can have a sort key as well - Can be defined even after table creation time – The index is essentially a new table that is created as an index table along with the main table –

Options for attribute projections –

KEYS_ONLY- new partition and sort key, old partition key and if applicable old sort key

    INCLUDE – specify custom projection values
    ALL – project all attributes

RCU/WCU is not shared with the main table and should be defined on the index

EMR

  • Basically EMR cluster is a collection of EC2 nodes (Master + Core Node + Task Node). Master manages the cluster – is mandatory – runs on EC2. Core Nodes runs tasks (data analysis) and hosts data (in HDFS or EMRFS). Task Nodes does not store data – utilized just for running tasks. Task Nodes can be utilized for scaling during peaks.

Note: EMR cluster is within a single AZ concept. Replicating data across AZs is tough!

  • Master Node: a.k.a “name node” Single master node – Manages the resources of the cluster – coordinates distributes and parallel execution of MapReduce executables – tracking and directing against HDFS in core nodes – Resource Manager (component of YARN) is a scheduler that manages as an example scheduling work across both MapReduce and Spark apps that might be running simultaneously – monitors health of core and task nodes
  • Core Node: a.k.a “data node” - Run tasks as directed – store data in HDFS (or in S3 using HDFS) – Data Node daemon runs on the core node – Node Manager / Application Manager work in tandem with the Resource Manager to manage, execute and monitor work that runs in the containers.
  • Task Node: similar to code node / but used only for compute / no storage (HDFS)
  • AWS EMR has advantages from standpoint of being able to easily integrate with other services. Major ones being -> EC2 for hosting, VPC, S3 (input/output data), CloudWatch, IAM, CloudTrail, Data Pipeline.

EMR Storage Detailed

  • HDFS: Typical Hadoop systems utilize HDFS – distributes data (files) across different nodes in the cluster – multiple copies on different nodes for redundancy (default replication factor is across 3 nodes – set in hdfs-site.xml file) – data is stored in blocks – large file gets split and stored across nodes (def block size is 64MB and max block size is 256MB). IMPORTANT -> Block sizes / Replication factor can be set per file.
  • Local / Instance Store or EBS: The files can be stores on the Instance Store volumes or EBS - both are ephemeral / transient and are cleaned in the cluster is terminated – Local Store is an option in cases very high I/O performance or high IOPS with low cost – EBS is not ephemeral but it is in this case where the cluster is terminated
  • EMRFS: access S3 as if it were HDFS. Looks like HDFS but is actually S3 – advantage is the data is not ephemeral anymore when cluster is terminated / resized. EMRFS and HDFS can be used together using S3DistCp command which is optimized to work with S3.
  • EMRFS Consistency View: when S3 is brought in the picture then also we will also need to ensure data is safeguarded against multiple concurrent access. This is where EMRFS Consistency View helps and it does the transaction mgmt. (i.e. consistency) utilizing DynamoDB. S3 PUTS have read-after-write consistency but if a list (HEAD/GET) command is executed right after PUT then the new objects might not show up. Case of deletes it is eventual consistency so deleted files might show up. EMRFS Consistency View resolves these as well as when multiple app access occurs.
  • S3DistCP: using an S3DistCp command is the right thing to do to copy data from S3 into HDFS and then make sure the data is processed locally by the EMR cluster MapReduce job. Upon completion, you will use S3DistCp again to push back the final result data to S3

  • Redshift Architecture:
  • Cluster: A cluster is the core infrastructure component of RedShift DW – consists of a ‘leader’ node and one or mode ‘compute’ nodes (max is 128) – each cluster can contain one or more databases – user data is stored on the computer nodes.
  • Leader node: acts as a gate with the SQL clients – mediator between clients and the compute nodes – receives queries and (parses queries + generates execution plans + coordinates with compute nodes to perform parallel execution + aggregates and return results)
  • Compute nodes: executes steps of the execution plan, coordinating with other compute nodes and finally sending results back to the leader node – each compute node has dedicated compute, memory and attached disk storage. Two types of compute nodes can be chosen during provisioning 1) Dense storage DS [HDDs] or 2) Dense Compute DC [SSDs]) – for both types there are large or 8xlarge choices.

Within compute node there are node slices – to process chunks of data – a portion of the compute mode CPU and disk space is allocated to each node slice

  • Redshift Table Design: Follows Star Schema which is the simplest style of data mart and is mostly widely used to develop data warehouses and data marts – The Star schema consists of one or more fact tables referencing any number of dimension tables. Other flavor is Snowflake schema design

Note: Transaction Performance Processing Council (TPCH) dataset is a standard / typically utilized to do performance benchmarks between databases / DWs

  • Redshift Distribution styles: how the data is distributed across the different compute nodes and slices – this is mentioned when creating the table – the two goals of distribution is to distribute the data evenly across the cluster and minimize data movement during query execution
  • EVEN: regardless of values leader node distributes ROWS in a round robin fashion – this is the DEFAULT distribution style
  • KEY: all data (i.e ROWS) are distributed based on matching columns / values – related/matching data are all stored physically closer. Example – if two tables are joined using a order_id columns the order id values in table1 and table2 are co-located in the same node slice / compute node
  • ALL: copy of the entire table is distributed across all nodes – takes longer to insert – is utilized only for data that does not change – for slow moving tables i.e. data that does not change much

Tables used in joins use KEY distribution – large fact tables in a star schema use KEY distribution – no joins / reduced parallelism and where KEY and ALL or not a clear choice use EVEN distribution – data does not change or small sized ( a few million rows) or no common distribution key then use ALL distribution style

  • RedShift Sort Keys: similar to indexes in relational DBs – when creating a table then one or more columns can be denoted as sort keys – when data is loaded then rows are stored in a sorted fashion so enables efficient handing of range restricted predicates….the query processor can use the min and max values can be utilized to rapidly skip over large number of blocks during scan

Example: If recent data is queried more often then timestamp column is chosen as the sort key | If frequent range or equality on a particular column then use that as the sort key | if tables are frequently joined then utilize the join column as the distribution and sort key

Sort Key Types:

  • Single (ex: querying primarily by date)
  • Compound (made up of more than one sort key – this is the default sort type – used in cases of conditions and joins – primary sort column and secondary sort column – order matters) – primary sort key has more weightage than secondary sort keys
  • Interleaved sort key (equal weightage to all columns in the sort keys) – great and use only on very large tables (>100 million rows) – think 3 index tables copies w/ each sorted by movie_id and movie_name and movie_genre
  • RedShift Compression: storing less data on disk → reduces cost. also reduces I/O for faster performance. Automatic Vs Manual way to compress – AWS recommends Automatic mode.

Automatic Mode: create DDL -> use COPY command to load data: behind the scenes -> data is loaded (first time) -> run analyze compression command -> determine types of encoding needed -> drop table -> recreate with encodings -> reload data

Manual mode: same as above but run command ‘analyze compression table_name’

Encryption

AWS IOT Security

  • Authorization: using AWS IOT policies (to/from AWS IOT services) and IAM policies (to/from AWS services).

AWS IOT policies (operations) is further divided across – control pane APIs for admin tasks and Data plane API for sending and receiving data from AWS IoT

IAM policies for users, groups and roles is also required when the AWS IOT services need to talk to other AWS services (exampe-> Rule action to send to firehose or write to DynamoDB)

KMS

  • preferred way for managing keys – utilized for encrypting data in EBS, S3, RedShift, RDS, SSM Parameter Store, etc – AWS manages the keys (customer master keys) – AWS does the encryption/decryption – KMS can only encrypt data that is max 4KB in size – for larger data utilize “Envelope Encryption” – can do only symmetric and not asymmetric encryption
  • Using KMS you can create keys, set rotation policies, disable keys, enable keys + use audit trail to check how keys are used using cloud trail.
  • 3 types of Customer Master Keys: AWS managed service default CMK / User keys created in KMS / User Keys imported - you pay for API calls to KMS ($0.03/10k calls)
  • When SSE-KMS is utilized with respect to S3 then the encryption/decryption occurs on S3 – If you want to utilize SSE-KMS for encryption/decryption within you applications then you utilize Encrypt API / Decrypt API

Cloud HSM

  • alternative for KMS – AWS just provides dedicated / secure encryption hardware and all software (encryption/decryption/key mgmt.) is upto the apps – the CloudHSM hardware device is tamper resistant – supports FIPS 140-2 Level 3 compliance – Cloud HSM clusters are spread across multiple AZs – supports both symmetric and asymmetric software (unlike KMS which does only symmetric)

EMR Security

  • EMR cluster can be run in a private subnet if security policies dictates such – in this case to access S3 then a S3 VPC gateway will also need to be defined
  • Encryption at rest / transit - For data encryption at rest on EBS/instance store 2 options exists -> Open Source HDFS / (Hadoop NVMe) encryption and LUKS (Linux unified Key Setp) encryption – for both of these options a KMS or custom key can also be configured and provided.   • Storing data in Hadoop using those raw formats is terribly inefficient. So options for efficient storage are typically Parquet, ORC or Avro. ORC, Parquet, and Avro are also machine-readable binary formats. Parquet and ORC both store data in columns, while Avro stores data in a row-based format.

Files stored in ORC, Parquet, and Avro formats can be split across multiple disks, which lends themselves to scalability and parallel processing. All three formats carry the data schema in the files themselves, which is to say they’re self-described.
